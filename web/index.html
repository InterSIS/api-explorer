Introduction

What is InterSIS?

InterSIS helps software developers and school districts hold student information in a single, secure place. InterSIS verifies the identity of students, staff, and guardians, and controls what information any of these users can see. InterSIS lets districts share this information with educational software applications.

InterSIS allows users to choose where to store sensitive student information: a school district could maintain their own InterSIS instance; a software developer could contract an experienced third-party to secure student information for districts that don't have InterSIS, or host an instance itself; a state government could host and secure sensitive student information for all of its public schools.

InterSIS includes software libraries that help developers easily access student information irrespective of where it is stored. And InterSIS is free and open source: it doesn't cost schools any money and it doesn't cost developers any money.

Technical Description: InterSIS is a backend for student information and educator information and authentication, using OAuth 2.0. InterSIS delivers resources over a RESTful API using JSON.

Software developers can use InterSIS to manage student data within their own applications, or receive permissioned access to a customer's instance if that customer already uses InterSIS.

This documentation is under development. The InterSIS specification and implementation are under development.

Some documentation sections may be missing or incomplete. Text decorated with line-through should be read as suggestive. Some features described in the documentation may not yet be implemented.

Who is this Guide for?

This guide is for educational software developers who would like to use InterSIS to retrieve or store student information.

Why use InterSIS?

  1. Reduce Development Overhead

    Class Definitions and ORM

    How many times have you written a class definition for a student? Our SDKs include class definitions and ORM for all of the data types—students, staff, sites, courses, sections, etc.— described below. If we have an SDK for your language or platform, integration with InterSIS can be as easy as executing a single command in your package manager:

    > pip install django-intersis

    After including InterSIS in your project, access objects in your own native ORM:

    these_students = this_course.students.all()

    Authentication

    InterSIS provides authentication services via OAuth 2.0. The SDKs include the basic InterSIS OAuth 2.0 workflow.

    Permissions

    Each request you make for InterSIS resources includes the identity of the end-user for whom you are making that request. InterSIS considers the privileges and relationships of that user in fulfilling the request. For example, a teacher will not be given biographical information for a student that they do not teach.

  2. Reduce FERPA liability

    Educational software developers who handle personally identifiable student information have a duty under the Federal Education Rights and Privacy Act (FERPA) to handle that information securely. In his 2015 State of the Union Address, President Obama affirmed a commitment to maintaining the security of digital student information. InterSIS authenticates all end-users via OAuth 2.0 and manages their access to sensitive information.

    Your software can request access to as much or as little personally identifiable student information as you require. Here's the response you might receive from /students/6ukarkb5wbuk5tbawrypdkexde/ with full read scope:

    {
  "id": "6ukarkb5wbuk5tbawrypdkexde",
  "hispanic_ethnicity": null,
  "race": null,
  "sites": [
    "lr4baqnj4ggaapzlexmlyvruw4"
  ],
  "sections": [
    "lr4baqnj4ggaapzlexmlyvruw4"
  ],
  "family_names": "Smith",
  "given_names": "Johnny",
  "birth_date": "1999-04-24",
  "sex": "M",
  "profile_name": "Grizzly84",
  "school_student_id": "248352",
  "state_student_id": "47832891038475",
  "guardians": []
}

    whereas software which had negotiated only the read-student-profile-names scope would receive:

    {
  "id": "6ukarkb5wbuk5tbawrypdkexde",
  "profile_name": "Grizzly84",
}

    The less information your software can access, the lighter your burden under FERPA.

  3. Reduce User Adoption Overhead

    Eliminate account creation and rostering from your user-adoption workflow. If a new teacher user has an existing InterSIS instance, they can grant your application privileges to their existing class roster and student accounts in seconds. When students log into your site, they authenticate via OAuth with the InterSIS instance too, and InterSIS will provide your application with the information needed to associate each student with their teacher's account.

Technical Definitions

This document may use the following definitions, from the OAuth 2.0 RFC 6749:

Term Definition Examples
resource owner An entity capable of granting access to a protected resource. When the resource owner is a person, it is referred to as an end-user. a teacher, student, administrator, or guardian
resource server The server hosting the protected resources, capable of accepting and responding to protected resource requests using access tokens. an InterSIS instance
client An application making protected resource requests on behalf of the resource owner and with its authorization. The term "client" does not imply any particular implementation characteristics (e.g., whether the application executes on a server, a desktop, or other devices). your web, iOS, Android, or desktop application
authorization server The server issuing access tokens to the client after successfully authenticating the resource owner and obtaining authorization. an InterSIS instance

Note that a single InterSIS instance serves the role of both resource and authorization server.

Authentication

Single Sign-On

New and returning users can "Sign on with InterSIS" and grant your application permissioned access to their student information and accounts in seconds.

For users without an InterSIS instance, your own InterSIS backend can manage authentication and across all of your applications, irrespective of their language or platform.

If you've already got experience with OAuth 2.0, or you're using a third party library to handle authentication, know that InterSIS does OAuth 2.0 a little different than other providers.

Unlike Facebook or Google, there's not just one InterSIS authentication server; anyone can run an InterSIS instance and host it anywhere. And instead of manually registering your client with an InterSIS instance, your client registers itself automatically. That's legal under the OAuth 2.0 spec, but you might not have seen it before. So if you already know your way around OAuth, read the following sections on Server Discovery and Client Registration.

Server Discovery

A user's InterSIS instance may live anywhere:

  • An SIS developer be using their own servers to host their customers' SIS instances.

  • A school district may host an InterSIS instance themselves to retain physical control of student data.

  • As a developer, you will have an InterSIS instance to provide storage and authentication for any users who do not yet have their data on an InterSIS instance.

When a new user with an existing InterSIS instance signs up for your services, they will need to tell you where that instance lives. It may live at intersis.exampleschooldistrict.org, or at esd.intersis.sisprovider.com, or any other url. If a new user does not have an InterSIS instance, then you will need to direct them to your own instance.

Client Registration

Once you know the location of a new user's InterSIS instance, your client must register with that server.

The OAuth 2.0 specification requires that clients register with the authorization server before directing end users to that server to authenticate. The endpoint at /v.1/auth/clients/ provides automated client registration with the authorization server. Your client must store the client credentials returned by that call.

During client registration, your client must provide the authorization server with a space-separated list of 'redirect-uris'.

For a school or SIS provider, automated client registration with the InterSIS authorization server allows their users to easily confirm their identity for any educational software that is integrated with InterSIS.

Those developers hosting an InterSIS instance to manage student data provided within their own software will generally not want to provide global authentication services for their users. These developers will wish to turn off automated client registration and register only their own applications manually. See your developer's guide for instructions.

You should only register with one InterSIS instance once. When a teacher returns to log in in a second time, or when a teacher's students visit the site to log in, you should be able to retrieve the client id and client secret you received the from the initial client registration.

User Authentication

A client which is registered with the authentication server may direct end users to authenticate at the endpoint /o/authorize/. Here's a (not yet encoded) URL that you might send your users to:

GET /o/authorize/?redirect_uri=https://myapp.example.com/intersis/login/callback/&scope=read-rostering read-student-names&state=7s3478d790JJd&response_type=code&client_id=KJDuer7#7010Dj278DFhgjk3321ahdjdjfwuriqoAjkLL

Talk about the state variable.

The following scopes are available for you to negotiate:

Scope Name Scope Description
read-writeRead/Modify/Create any resources that the end user has permission to read/modify/create.
readRead any SIS information that the end user has permission to access.
read-rosteringRead the end user's course names, course descriptions, and section rosters, not including student names or demographic information.
read-student-namesRead any student names and profile names the end user has permission to access.
read-student-profile-namesRead any student profile names the end user has permission to access.

Note that if a client is not granted read access to student names, the resource server will still provide the client with a unique ID string that may be used to distinguish among students. The ID string which identifies a student-as-resource is identical to the ID string which identifies that student as an end user.

Client Type Use Appropriate Scopes
Student Information System Frontend Teachers will need access to all rostering and demographic information, as well as the ability to enter grades.
School administrators will need to be able to add and remove students to their sites as well as to individual sections.
Guardians will need to be able to read their students' grades		 read-write
Skills Practice Web Client Teachers will need to assign tasks to and monitor the performance of their students, by name. Students will require the site to track and display their progress. read-rostering,
read-student-names
Educational Flash Game Site Students will require the site to keep track of their game scores. read-student-profile-names

User Identification

Once the client has received an access token, it may request the identity of the end user at the endpoint '/v.1/auth/user/'

Subsequent Visits

When a user returns for subsequent visits, you should be able to retrieve the client id and client secret that you stored when registering your client with that user's InterSIS instance.

          Development Roadmap

          InterSIS is currently in development of version .70 of the InterSIS API.

          Version .70:

          Version .70 provides all of the features described above for storing student information, and providing end user authentication via OAuth 2.0.

          Version .75:

          Following the release of of version .70, the InterSIS Foundation will develop the SDKs and test integration. Those API features found necessary during this process will be added to version .75.

          Version .80:

          Version .80 will include the ability for application clients to write gradebook items and scores to the InterSIS server.

          Version 1.0:

          Following the release of version .80, we will freeze the InterSIS API specification and focus on hardening the Django implementation against attack. Version 1.0 will be announced when this hardening is complete. While awaiting version 1.0, developers will be able to take advantage specification freeze to integrate InterSIS into their clients.

          Version 2.0:

          Following the completion of version 1.0, we will seek feedback from our partner developers. The bulk of modifications to the InterSIS API specification will come from insights gained from this feedback.

          Version 2.0 will also include the capacity for InterSIS to act as an authentication server for resources housed on third-party resource servers.

          Version 3.0:

          Features included in version 3.0 of the InterSIS API specification will come from feedback from the greater developer community.
